Master Instruction for LLM: Build the Compliance Gap Analyser
You are an expert full‑stack developer, actuary, and DevOps engineer. Your task is to create a Streamlit application for Australian life insurance model governance gap analysis. The app must:

Pose a structured questionnaire to the user about their model governance lifecycle.

Map answers to specific clauses/requirements from APRA prudential standards (CPS 230, CPS 220, CPS 510, LAGIC standards like LPS 110, LPS 220, etc.) and Actuaries Institute Professional Standards (e.g., PS 200, PS 300 series, life insurance valuation standards).

Identify compliance gaps and generate a gap report with severity, evidence of standard, and concrete mitigation actions.

Be fully extensible: later we will attach a REST API and an LLM (e.g., via LangChain) that can auto‑analyse a company’s documents and answer the same questionnaire.

Crucially, the system must automatically fetch and update the source standards from official websites, process them, and store them locally in a form that supports both rule‑based gap detection today and semantic LLM queries tomorrow. Use a local vector database (ChromaDB) with embeddings and a structured JSON store for the gap rules.

2. Detailed Project Workplan (Phases)
Phase 1: Requirements & Knowledge Curation (Manual Bootstrapping)
Identify all relevant Australian standards

APRA: CPS 230 (Operational Risk), CPS 220 (Risk Management), CPS 510 (Governance), LPS 110 (Capital Adequacy), LPS 220 (Risk Management for Life Insurers), etc.

Actuarial: Professional Standard 200 (Life Insurance Valuation), PS 300 (Actuarial Reporting), PS 400 (Model Governance – if applicable), and any relevant Guidance Notes.

Curate a master structured rule set (initially JSON)

For each standard, break down into granular requirements (e.g., “CPS 230 ¶ 27(b): documented risk appetite statement”).

Design a questionnaire that probes these requirements.

Map each question to one or many requirement IDs; assign a gap logic (e.g., if answer is “No” → gap).

For each gap, write a mitigation recommendation (text) and link to the exact clause.

This will be stored in data/gap_rules.json.

Source URLs for standards (APRA website, Actuaries Institute website). Record them in a standards_sources.yaml.

Phase 2: Standards Ingestion Pipeline (Automatic)
Build a standalone module standards_ingestion that:

Reads standards_sources.yaml (URL, title, category, expected last-modified date).

Downloads PDFs to data/raw_pdfs/ using requests with polite headers and ETag/Last-Modified checking to avoid redundant downloads.

Parses each PDF with PyMuPDF (fast text extraction) → doc_id, source, version, timestamp.

Splits text into overlapping chunks (e.g., 500 tokens with 50 overlap) using RecursiveCharacterTextSplitter.

Generates embeddings using sentence-transformers/all-MiniLM-L6-v2 (384 dims).

Stores chunks in ChromaDB collection standards_collection with metadata: standard_name, clause (if detectable via regex), chunk_index, published_date, source_url.

Optionally attempts to extract clause numbers via regex (e.g., ¶ 27(b), Paragraph 30) and stores them for direct clause lookup.

Logs the process and saves a last_update.json with timestamps.

This pipeline can be run on app startup (if out of date) or scheduled via APScheduler inside Streamlit (or via a separate CLI script for maintenance).

Phase 3: Structured Rule Base & Questionnaire Design
Design the questionnaire JSON (or Pydantic models) with:

Sections (e.g., “Board Oversight”, “Model Development”, “Model Validation”, “Data Governance”, “Documentation”, “Independent Review”).

Questions with answer type: boolean (Yes/No), multi-choice, text.

Each question links to one or more requirement_ids.

Each requirement has conditional gap logic: {"if_answer": "No", "gap": true, "severity": "high", "mitigation": "..."}.

Store the full definition in data/questionnaire.json and data/gap_rules.json.

Phase 4: Gap Analysis Engine
Module engine/gap_analyzer.py:

Takes the user’s answers (dictionary).

Evaluates all mapped requirements.

Computes a list of GapFinding objects: requirement_id, clause_reference, question, user_answer, gap_severity, mitigation, evidence_text (optional snippet from vector store).

For each finding, optionally performs a similarity search in the vector store to retrieve the relevant standard text, to show the user the exact source.

Phase 5: Streamlit UI
Multi‑page app (using st.navigation or session state page switching):

Home: Introduction, disclaimer, option to start new analysis or upload previous session.
Questionnaire: Stepped form, progress bar, dynamic sections.
Gap Report: Interactive table (filter by severity, category), expandable evidence with standard text. Export to PDF/Excel.
Admin (hidden): Trigger standards update manually, view logs, check vector store status.
All heavy processing cached via @st.cache_resource (vector store, rule engine).

State management via st.session_state (answers, current step).

Phase 6: Future API & LLM Integration
Design a separate api module using FastAPI that exposes:

POST /analyze – receives a set of answers and returns gap report JSON.

GET /standards/search?q=... – semantic search over standards.

POST /async_analyze – accepts uploaded documents, triggers LLM‑powered analysis.

The vector store and rule engine are shared, so the API can run as a sub‑application in the same process (or separate container).

For LLM integration, later we will build a retriever from ChromaDB and use an LLM chain to answer the questionnaire automatically given a company’s policy PDFs.

4. Coding Standards & Conventions (Mandatory)
Python style: black (line length 100), ruff for linting, mypy strict mode.

Type hints: Every function signature and complex variable must be annotated.

Docstrings: Google style for modules, classes, and functions; include Args, Returns, Raises.

Configuration: All configurable values (file paths, model names, API keys) in config.py using pydantic.BaseSettings, loaded from environment / .env.

Logging: Use logging module, configure in app.py; do not use print for operational messages.

Error handling: Never swallow exceptions silently. Use custom exception classes (StandardsDownloadError, IngestionError). Streamlit UI catches and displays user-friendly errors.

Modularity: Keep components loosely coupled; the engine must be importable and testable without Streamlit.

Testing: Unit tests for gap logic, PDF parser (with fixtures), embeddings (mock ChromaDB). Aim for >80% coverage.

Version control: Use Git; the data/gap_rules.json and data/questionnaire.json are part of the repo (they are curated). data/raw_pdfs/ and data/chroma_db/ are in .gitignore.

5. Automatic Standards Update & Local Storage Efficiency
The “efficient local storage” is achieved by:

Vector store (ChromaDB) – persists on disk, supports incremental updates (upsert by doc_id). Embeddings are generated once per chunk and reused; only changed PDFs re‑indexed.

Smart downloading – uses HTTP If-Modified-Since and ETag to avoid re‑downloading unchanged files. A metadata file last_update.json tracks last successful fetch per source.

Modular pipeline – can be triggered via:

Streamlit admin button (synchronous, using st.cache_resource to avoid re‑ingestion).

A scheduled background thread using APScheduler that runs once a week (if Streamlit is running).

A separate CLI script python -m scripts.run_ingestion that can be called by external cron or a GitHub Action.

Chunking & embedding strategy:

Use sentence‑transformers with a small, fast model (384‑dim) to keep the vector index compact.

Split only at paragraph boundaries when possible; don’t break mid‑clause.

Store clause numbers as explicit metadata (via regex extraction during parsing) so that gap rules can directly look up “CPS 230 ¶ 27(b)” without a fuzzy search.

def get_evidence_text(requirement_description: str, collection, embedder, k=1) -> str:
    query_embedding = embedder.embed_query(requirement_description)
    results = collection.query(query_embeddings=[query_embedding], n_results=k)
    return results['documents'][0][0] if results['documents'] else ""
Gap rule evaluation:

python
def evaluate_rule(rule: GapRule, answers: dict) -> bool:
    user_answer = answers.get(rule.question_id)
    if user_answer is None:
        return False
    if rule.gap_condition.logic == "equals":
        return user_answer == rule.gap_condition.value
    # add more logic types (contains, not_equals)
    return False

9. Maintenance and Evolution Plan
Standards update monitoring: Regularly check APRA and Actuaries Institute sitemaps. The sources.yaml can be extended with new standards.

Gap rules evolution: The gap_rules.json is version‑controlled; any changes to questionnaire or rules require a review by a qualified actuary. The system can include a “rule version” to flag analyses performed under older rules.

LLM readiness: When integrating an LLM, the gap_analyzer can accept answers that were auto‑generated by the LLM after reading documents. The engine remains unchanged. The vector store will serve as the retrieval knowledge base for the LLM (via LangChain’s Chroma retriever).

API first: The FastAPI layer will be built so that the same engine can be used headless, enabling batch analysis or integration into CI/CD pipelines for model governance.
